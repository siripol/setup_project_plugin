# Claude security guidance — ${name}

In-session security rules the Claude review applies to every diff before it is committed. These rules are the hook point where the org compliance pack attaches when enabled.

This file is the **per-repo, in-session review checklist** for Claude when it generates or edits code in `${name}`. For template-family-level security posture (baseline controls, incident-class reasoning, pinning policy, threat-model boundary, escalation), see `docs/SECURITY.md`. The two files are complements: `docs/SECURITY.md` describes the controls; this file describes what Claude must check while writing the code those controls run on.

## What Claude checks before suggesting a Write/Edit

Every diff Claude proposes in this repo runs through the rules below. If a rule trips, Claude pauses and surfaces the concern to the user before completing the operation.

### Secrets and credentials

- No literal API keys, tokens, passwords, or private-key material in source, tests, fixtures, or docs.
- No credential strings in commit messages, comments, or example URLs (use `EXAMPLE_TOKEN`, not the real shape).
- Sensitive configuration goes in `.env.example` (committed, no values) plus `.env` (gitignored, values).
- Test fixtures use deterministic fake values, never anonymized real values.

### Input validation at trust boundaries

- All external input (HTTP request bodies, query strings, message-queue payloads, user-uploaded files) is validated at the boundary handler.
- Validation rejects unknown fields by default; do not silently drop them.
- Numeric inputs have explicit bounds; string inputs have explicit max length.
- Path inputs are normalized and checked against directory-traversal patterns (`../`, absolute paths).

### AuthZ at every endpoint

- Every external endpoint declares an explicit authorization policy.
- No endpoint defaults to "anyone authenticated"; the policy is named in code.
- Object-level authorization checks happen before any object access (avoid IDOR via "scope by query filter alone").
- AuthZ failures emit a single audit log line, never a 500 with stack trace.

### Output handling

- No raw user input echoed back in error messages, logs, or audit entries unless explicitly sanitized.
- Logs use structured fields, never string-interpolated SQL or shell commands.
- HTML / template rendering uses the framework's auto-escape; if it must be disabled, the disable scope is one line and reviewed.

### Dependency hygiene

- No new dependency introduced without a one-line justification in the PR description.
- Dependencies pin to specific versions in lockfiles (committed).
- License compatibility check passes (most permissive set: MIT / BSD / Apache-2; copyleft requires opt-in).

### Audit and observability

- New code paths emit at least one structured log line at info level on success and error.
- Sensitive operations (login, role change, regulated-data read, regulated-data write) emit a dedicated audit log line distinct from operational logs.
- Audit log fields are field-named, never positional. Field names are stable across releases.

## Compliance pack attachment point

When the `pdpa-compliance` policy (or any future compliance pack) is applied via `sn-setup policy apply <slug>`, additional in-session review rules attach **here**. The plugin extends this file with the pack-specific rules during apply; the body above is the base set every repo starts from.

For PDPA specifically, the attached rules add:

- PII detection patterns (Thai NI, email, phone, PAN, passport) — Claude flags any Write/Edit landing such patterns outside the allowlist.
- Retention sidecar verification on Writes to `data/` — Claude verifies the sidecar exists and is current before suggesting the write.
- Consent record check on Writes to `data/subjects/<id>/**` — Claude verifies the matching consent record exists with valid `purposes` + `expires_at`.

These extensions surface as comments inserted between the `## Compliance pack attachment point` heading and the `## See also` footer.

## Override

If Claude trips a rule the user knows is acceptable (e.g. seeding a deliberate test fixture):

1. The user states the override explicitly: "this is intentional, skip the secret-scan for this fixture".
2. Claude proceeds but adds a comment `# claude-security: intentional fixture, see PR #N` adjacent to the change.
3. The PR reviewer sees the comment and decides whether the override holds.

Do not silently bypass. Do not edit this file to remove rules the team disagrees with; open a PR to discuss the change first.
